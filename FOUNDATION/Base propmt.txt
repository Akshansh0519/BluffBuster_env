════════════════════════════════════════════
2. STUDENT SIMULATOR FAMILY (CRITICAL — DO NOT SIMPLIFY)
════════════════════════════════════════════

The student simulator is NOT a single scripted agent. It is a FAMILY of styles with PARAMETRIZED LEAK RATES, sampled per episode and per section. Leak rates are the critical knob that gives the diagnostic reward signal something real to detect — without them, fakers would be trivially detectable and the RL signal would be meaningless. Tune leak rates to make the discrimination task non-trivial but learnable.

KNOWS Styles (3 modes — one sampled per section that is in the KNOWS partition):
  K1: mechanistic-clear        — uses causal language, names specific mechanisms, gives concrete examples
                                 mechanism_cue_emit_rate: 0.85   misconception_emit_rate: 0.05
  K2: concise-expert           — brief accurate answers, correct but terse
                                 mechanism_cue_emit_rate: 0.55   misconception_emit_rate: 0.05
  K3: uncertainty-aware-expert — hedges appropriately ("I'm less sure about X but..."), still correct
                                 mechanism_cue_emit_rate: 0.65   misconception_emit_rate: 0.08

FAKING Styles (4 modes — one sampled per section that is in the FAKING partition):
  F1: definitional-bluffer     — recites textbook definitions correctly, collapses under "why/how/edge-case" probes
                                 mechanism_cue_emit_rate: 0.15   misconception_emit_rate: 0.30
                                 collapse_under_mechanism_probe: 0.80  (probability the bluff visibly degrades when probed)
  F2: jargon-matcher           — mirrors technical terms from the question; plausible-sounding but empty
                                 mechanism_cue_emit_rate: 0.20 (parroted, low-context)   misconception_emit_rate: 0.25
                                 jargon_mirror_rate: 0.70
  F3: evasive-generalizer      — pivots to a safer adjacent topic; subtle subject change
                                 mechanism_cue_emit_rate: 0.10   misconception_emit_rate: 0.20
                                 topic_drift_rate: 0.60
  F4: overconfident-handwaver  — very confident, very high-level, zero specificity; "this is basically just X"
                                 mechanism_cue_emit_rate: 0.05   misconception_emit_rate: 0.40
                                 specificity_score_cap: 0.20

Leak rate semantics: each rate is the probability the corresponding token-class appears in a single response, modulated by the question's probe structure. A "why/how/edge-case" probe to a faker downshifts mechanism_cue_emit_rate by 50% and upshifts misconception_emit_rate by 50% — this is what creates the diagnostic gap that R_info can detect.

Style Assignment:
- Style sampled at episode reset using episode seed for reproducibility
- Each section in KNOWS partition gets a KNOWS style; each section in FAKING partition gets a FAKING style
- Per-section style is sampled independently; the same student can be K1 on S01 and K3 on S02
- Style is never revealed to the examiner
- Eval suite uses a HELD-OUT style combination that the examiner never sees during training (specified in eval_config.json)

StudentProfile Object (required in implementation, Pydantic v2):
  knowledge_mode: Literal["KNOWS", "FAKING"]
  style: Literal["K1","K2","K3","F1","F2","F3","F4"]
  section_id: str  (S01–S10)
  verbosity: Literal["brief","medium","verbose"]   (sampled)
  confidence_pattern: Literal["hedging","neutral","confident"]   (style-driven)
  mechanism_cue_emit_rate: float  ∈ [0,1]
  misconception_emit_rate: float  ∈ [0,1]
  style_specific_params: dict     (collapse/mirror/drift/cap fields above)
  seed: int                       (for reproducibility — same seed = byte-identical response)

Student responses MUST be assembled deterministically from KB fields:
- KNOWS templates draw from KB[section].mechanism_cues and KB[section].key_concepts
- FAKING templates draw from KB[section].common_misconceptions and lexically-similar-but-shallow phrasings
- Probe sensitivity: the response generator inspects question_text for probe markers (why/how/what-if/edge-case/counterexample) and applies the leak-rate modulation above
- All sampling uses a per-(episode_seed, turn, section) RNG — reproducible across runs

════════════════════════════════════════════
6. REWARD FUNCTION — INFORMATION-GAIN COMPOSITE (PROGRAMMATIC, BOUNDED)
════════════════════════════════════════════

DESIGN PHILOSOPHY:
The reward must mirror the scientific claim — the examiner is rewarded for asking questions whose answers carry the most information about the hidden KNOWS/FAKING partition, and for committing to a CALIBRATED final classification. We use:
  (a) a sparse terminal reward (correctness + asymmetric error costs + calibration margin),
  (b) a potential-based dense shaping signal (per-turn information gain), which by Ng et al. (1999) does NOT change the optimal policy,
  (c) hard step-level penalties for malformed/repeated/invalid actions.

All components are programmatic, deterministic given (seed, KB, dialogue), bounded, and computed without any LLM judge or token-overlap heuristic. Reward is finalized at episode end. Step-level penalties fire immediately AND accumulate into the terminal reward for logging.

────────────────────────────────────────────
6.1 KB-GROUNDED POSTERIOR ORACLE  (used ONLY for shaping — never observed by examiner, never used as ground truth)
────────────────────────────────────────────

For each section s, the oracle maintains a binary posterior p_t(s) = P(KNOWS | history_t).

After turn t in which the examiner asked question q about section s and received response r, the oracle updates:

  evidence(r, s) = α · mechanism_cue_coverage(r, KB[s]) − γ · misconception_count(r, s)
  relevance(q, s) = β · probe_strength(q, KB[s].probe_templates)
  LLR_t(s)        = relevance(q, s) · evidence(r, s)         (clipped to [−3, +3])

  posterior_t(s)  = sigmoid( logit(p_{t−1}(s)) + LLR_t(s) )

Sections not asked this turn keep their prior. Initial p_0(s) = 0.5 ∀ s.

Constants α, β, γ are loaded from KB[s].evidence_weights (defaults 1.5, 0.5, 1.0).
Calibrate constants on a 200-episode held-out KB calibration split BEFORE training begins; store calibrated values in outputs/eval/oracle_calibration.json. Target: oracle posterior at t=MAX_TURNS achieves ≥75% accuracy on calibration split with mean Brier score ≤0.18.

The oracle is a deterministic surrogate, NOT a true Bayesian oracle. It is used only to (i) shape R_info and (ii) compute R_cal. It is never used to grade final accuracy — that uses true_label.

────────────────────────────────────────────
6.2 REWARD COMPONENTS  (all bounded, weights explicit)
────────────────────────────────────────────

──── Terminal components (computed on Classify) ────

R_acc   = (1/N) · Σ_s [ 1 if classification[s]==true_label[s] else −1 ]                 ∈ [−1, +1]
          where N = number of sections classified (typically 10)

R_asym  = − (λ_FA · #false_accusations + λ_FE · #false_exonerations) / N                ∈ [−0.5, 0]
          λ_FA = 0.5  (KNOWS labeled FAKING — costlier; punishing genuine students)
          λ_FE = 0.3  (FAKING labeled KNOWS — letting bluffers pass)
          Asymmetry encodes the project's value judgment that false accusations are worse.

R_cal   = (0.4/N) · Σ_s sign(correct(s) − 0.5) · |2·p_T(s) − 1|                          ∈ [−0.4, +0.4]
          Rewards confident-correct (high posterior margin AND right) and punishes confident-wrong.
          Implements calibration without requiring the model to emit explicit probabilities.

R_eff   = 0.20 · max(0, (MAX_TURNS − turns_used) / MAX_TURNS) · 𝟙[R_acc > 0]            ∈ [0, +0.20]
          Efficiency only earned when accuracy is positive. No reward for fast-wrong answers.

R_cov   = − 0.30 · 𝟙[any required section missing from classify]
          − 0.05 · (#missing_sections / 10)                                              ∈ [−0.35, 0]

──── Shaping components (potential-based; aggregated at episode end) ────

R_info  = 0.40 · clip( Σ_t ΔH_t , 0, 1 )                                                 ∈ [0, +0.40]
          ΔH_t = H(p_{t−1}) − H(p_t)
          H(p) = − (1/N) · Σ_s [ p_s·log p_s + (1−p_s)·log(1−p_s) ]   (mean per-section binary entropy, log base 2)
          ΔH_t is per-turn entropy reduction summed over the episode and clipped to [0, 1] before scaling.
          This is the PRIMARY diagnostic signal. Rewards questions whose answers measurably move the oracle posterior toward truth. Potential-based: total = φ(p_T) − φ(p_0) where φ = −H, so optimal policy is unchanged vs. terminal-only reward.

R_qual  = 0.10 · mean_over_asks( question_features(q) )                                  ∈ [0, +0.10]
          question_features ∈ [0, 1] is question-side only (does not depend on response):
            0.40 · 𝟙[mechanism_probe_present]   (why/how/mechanism keywords or KB cue match)
          + 0.30 · 𝟙[specificity_demand]        (asks for example, edge case, or quantitative answer)
          + 0.30 · 𝟙[edge_case_or_counterexample]  (boundary, failure mode, comparison probes)
          Pure prior on good questioning style. Decoupled from response so it can't be gamed by friendly fakers.

R_div   = 0.05 · ( #unique_sections_asked / min(turns_used, N_sections) )                ∈ [0, +0.05]
          Lightweight diversity bonus; prevents collapse onto one section.

──── Hard penalties (per-step; sum into terminal reward) ────

P_malformed   = − 0.20 · #malformed_outputs                                              ≤ 0
P_repetition  = − 0.10 · #near_duplicate_asks   (cos_sim > 0.85 to prior ask in same section, sentence-transformer or KB-cue Jaccard fallback)   ≤ 0
P_invalid_sec = − 0.10 · #asks_with_invalid_section_id                                   ≤ 0

──── Total ────

R_total = R_acc + R_asym + R_cal + R_eff + R_cov + R_info + R_qual + R_div
        + P_malformed + P_repetition + P_invalid_sec

THEORETICAL BOUNDS:  R_total ∈ [ −2.05 ,  +1.95 ]
EXPECTED EMPIRICAL RANGE (DEMO config, after 200 eps): random ≈ −0.3, definitional ≈ +0.1, trained target ≥ +0.6
Random-rollout bounds are recomputed at training start and stored in outputs/eval/reward_bounds.json. Trainer will warn if observed reward exceeds theoretical bounds (indicates a bug).

────────────────────────────────────────────
6.3 GRPO INTEGRATION
────────────────────────────────────────────

The environment returns RAW bounded R_total. NEVER pre-normalize inside the env — the trainer owns advantage normalization.

GRPOTrainer applies group-normalization per prompt across K=num_generations rollouts:
  A_i = (R_i − μ_group) / (σ_group + 1e−6)
  A_i is then clipped to [−5, +5] before policy update (advantage clipping for stability).

KL coefficient: β_kl = 0.04 (lower than default to allow exploration, higher than 0 to stay close to base model). Increase to 0.1 if reward variance > 1.0 per group.

Reward variance budget per batch: target σ(R) ∈ [0.2, 1.0]. If σ(R) < 0.05 over 50 episodes, the run is degenerate (reward signal collapsed) — trainer logs a warning and Validator escalates.

────────────────────────────────────────────
6.4 REFERENCE CACHE
────────────────────────────────────────────

Pre-computed at environment init for each (section_id, probe_type) pair from KB.
Stores: (i) genuine baseline answer fingerprint (mechanism_cue set, length, specificity score),
        (ii) per-style expected leak fingerprint (used to validate simulator calibration only — not in reward).
Cache is deterministic, reproducible via seed, and saved to outputs/eval/reference_cache.pkl.
NEVER exposed to the examiner. NEVER used as a target for token-overlap (that approach is REMOVED — see §12).

────────────────────────────────────────────
6.5 REWARD HACKING MITIGATIONS (audited by Validator)
────────────────────────────────────────────

- Always-KNOWS / always-FAKING: R_acc + R_asym strictly negative on any mixed-truth episode; R_cal also collapses since posterior never moves away from prior 0.5.
- Single generic question repeated: P_repetition fires; ΔH ≈ 0 → R_info ≈ 0; R_qual ≈ 0; R_div ≤ 0.05.
- Asking one section, classifying others blindly: R_info only earned for asked sections; R_cal ≈ 0 on un-asked sections; R_cov small but R_acc dominated by guessing.
- Ending immediately at turn 0: R_eff = 0 (gated on R_acc>0), R_info = 0, posteriors at prior → R_cal ≈ 0, R_acc near random.
- Parser exploit (gibberish output): P_malformed fires every step.
- Style memorization: held-out style on eval suite collapses R_info gains; this is the generalization metric we report.
- Jargon-mimicking examiner (mirroring fake style): R_qual is question-side only and rewards mechanism cues, not jargon density.
- Posterior-gaming via leading questions: relevance(q, s) caps R_info contribution per turn; LLR clipped to [−3, +3] prevents single-turn explosion.
- Calibration gaming (always confidently right): only possible if oracle is well-calibrated, in which case it IS the desired behavior (not gaming).

8. EVALUATION ARCHITECTURE (SEPARATE FROM TRAINING)
════════════════════════════════════════════

Frozen Eval Suite (created ONCE before training begins, never modified):
  - 50 episodes with fixed seeds
  - Uses HELD-OUT fake styles: eval suite uses F1+F2 styles only during training, then tested on F3+F4 (or a defined held-out combination — specify in config)
  - Uses HELD-OUT sections: 2 sections (e.g., S09, S10) are never used in training episodes
  - Seeds are fixed and stored in eval_config.json

Evaluation runs:
  - Before training: run all 3 baselines on frozen eval suite → store results in outputs/eval/baseline_metrics.json
  - During training: evaluate trained checkpoint every N steps on frozen eval suite (not on training rollouts) → store in outputs/eval/checkpoint_metrics.json
  - After training: final eval on frozen eval suite → store in outputs/eval/final_metrics.json

Metrics computed per eval run (all logged per-episode and aggregated):
  - classification_accuracy            (per section and overall)
  - false_accusation_rate              (KNOWS classified as FAKING)
  - false_exoneration_rate             (FAKING classified as KNOWS)
  - avg_turns_to_classify              (efficiency proxy)

12. WHAT IS EXPLICITLY REMOVED FROM OLD PLAN
════════════════════════════════════════════

These items existed in a previous version of this project. They are REMOVED. Do not implement them. Do not reference them. Do not generate prompts for them.

REMOVED — Adaptive difficulty (difficulty scaling every 50 episodes):
  Reason: introduces confounding variables, makes eval less interpretable, not needed for hackathon.

REMOVED — Episode-index transcript selection (episode 10 vs episode 400):
  Reason: brittle, dishonest. Replaced by behavior-quality-based selection (see section 10).

REMOVED — Hard assertion that 10 GRPO steps must show reward improvement:
  Reason: unrealistic for short smoke tests. Replaced by pipeline health checks.

REMOVED — "3 visible training phases" as a hardcoded narrative:
  Reason: scripted storytelling that may not match reality. If real phases emerge from data, describe them. If not, do not narrate them.

REMOVED — Token-overlap divergence as the diagnostic quality metric:
  Reason: brittle to prompt/wording changes, not grounded. Replaced by KB-grounded posterior LLR + information-gain shaping.

REMOVED — Single generic fake student:
  Reason: examiner would overfit to one caricature. Replaced by 4 fake styles with parametrized leak rates.

REMOVED — "World's first" and "traditional benchmarks are dead" language:
  Reason: overclaiming. Replaced by precise scientific claim (see PROJECT IDENTITY section).

REMOVED — Single 7B full training as default/only config:
  Reason: likely to OOM on Colab without staged approach. Replaced by DEBUG/DEMO/FULL tier configs.

REMOVED — Symmetric ±1 per-section accuracy reward without calibration or asymmetric error costs:
  Reason: doesn't reflect real-world cost of false accusations vs. false exonerations and gives no signal for confident-wrong vs. confident-right behavior. Replaced by R_acc + R_asym + R_cal triple.

REMOVED — Step-level dense reward shaping NOT tied to information theory (e.g., per-step "good question" bonuses based on heuristic keyword counts):
  Reason: introduces non-stationary objectives, can be gamed, and biases the optimal policy. Replaced by potential-based R_info using oracle entropy reduction (theoretically guaranteed not to bias optimal policy).

REMOVED — In-environment reward normalization (z-scoring inside step()):
  Reason: corrupts GRPO advantage computation and hides reward bugs. Trainer owns normalization; environment returns raw bounded reward.

REMOVED — Using the posterior oracle as ground truth for accuracy:
  Reason: the oracle is a deterministic surrogate, not truth. Accuracy is always graded against the hidden true_label partition; the oracle is shaping-only.

Complete merge procedure (10 steps, no ambiguity):
1. Feature branch passes local sanity checks (3 conditions from task definition)
2. Open PR → assign Validator as reviewer
3. Validator runs AI code review prompt
4. Validator runs gate-specific MSR checks
5. Validator runs RL-specific manual checks for this gate
6. Any 🔴 finding blocks merge — fix required, re-review
7. Any 🟡 finding logged in mistakes.md — merge proceeds with note
8. Validator approves PR
9. Squash merge to main with structured commit message
10. All 3 team members pull latest main
